Open World Context Engineering Manual

This document explains how Dream-E's Open World mode constructs the AI context, what the scene-writing API returns, how validation works, and the rules governing each step.

1. Architecture Overview

The pipeline has 4 stages:

  1. Context Construction (openWorldContext.ts) — Assembles system prompt + user message from game state
  2. API Call (vite.config.ts /api/open-world) — Sends to Gemini or OpenAI-compatible API with streaming
  3. Response Parsing (openWorldService.ts) — Parses structured JSON, validates fields, enforces assertions
  4. Scene Creation (AdventureEngine.tsx) — Creates persistent SceneNode + StoryEdge, triggers image/music/TTS

3. Context Construction Rules

The context is split into two API message roles:

  • System prompt — From WriterSettings.systemPrompt (user-editable in AI Settings). Defines narrator personality, writing style, output format, and rules.
  • User message — Assembled procedurally from game state. Contains all context the model needs.

3.3 Compression Rules (Procedural)

  1. Below 200K tokens: Include everything. All scenes in full, all entity profiles.
  2. 200K–250K tokens: High-compression mode activates:
    • Story timeline: only last 3 scenes in full detail; older scenes use summary (1-2 lines each)
    • Low-priority entity profiles (not in current scene) are dropped — summaries remain
  3. Above 250K tokens: Further truncation:
    • Timeline keeps shrinking (minimum 2 full scenes)
    • All non-scene entity profiles dropped
    • Hard character limit applied (1M chars) with truncation warning appended

Never dropped: Project notes, entity summaries, current scene, variables, player action, instruction.

3.4 Story Timeline Construction

The timeline follows session history (array of visited node IDs). For each scene:

  • Full detail: Scene label, speaker name, full story text, choice made
  • Compressed: Scene label, summary (if available), choice made (1 line)

Scenes are always in chronological order. The "choice made" shows which option the player picked to reach the next scene.

3.5 Entity Categorization

Entities are split into two groups based on the current scene's linked fields:

  • Current-scene entities (from linkedCharacters, linkedLocations, linkedObjects, linkedConcepts) — get full profiles at HIGH priority
  • Other entities — get full profiles at LOW priority (dropped first under budget pressure)

ALL entities always get a one-line summary in the [ALL ENTITY SUMMARIES] section.

4. Structured JSON Output (Schema)

The API enforces structured JSON output. The model MUST return a JSON object. The flow analysis fields come FIRST — the model must reason about player engagement and narrative strategy before writing any scene content. This is enforced by schema ordering and the system prompt.

4.1 Flow Analysis Fields (filled FIRST — drive the scene)

These 5 fields are required and placed at the top of the schema. The model commits to a plan before writing.

FieldTypeRequired?Description
playerGoalHypothesisstringYesOne sentence: what does the PLAYER (not the character) want to experience? Action, puzzles, social drama, atmosphere, plot density? What topics and tone?
sceneIntentHypothesisstringYesOne sentence: what is the player trying to achieve with THIS specific action? What direction are they pushing the story?
lastSatisfactionEstimatestringYesOne sentence: how many scenes ago did the player last get what they wanted — a moment of success, fun, or the experience they seek? Reference the scene if possible.
engagementStrategystringYesOne sentence decision: should this scene SATISFY the player (reward, progress, fun) or CHALLENGE them (complication, conflict, uncertainty requiring active decision-making)? If not satisfied recently → lean toward satisfaction. If too easy → introduce challenge.
plannedStateChangesstringYesOne detailed sentence: which specific entities should change state, and how, to realize the engagement strategy? Changes must be plausible, emotionally intelligent, not cliche.
Chain-of-thought by design: These fields force the model to reason about player psychology and narrative strategy before generating the scene text. The schema places them first in the output, so the model must commit to a plan before writing. This improves scene quality, pacing, and responsiveness to player intent.

4.2 Scene Content Fields (written AFTER analysis)

These fields are driven by the flow analysis above. The sceneText realizes the engagement strategy; entity updates reflect the planned state changes.

FieldTypeRequired?Description
sceneTextstringYesNarrative continuation, 100-300 words, screenplay-inspired style. Driven by the analysis above.
speakerNamestringYesMain narrator/speaker name (default "Narrator")
choicesstring[]Yes3-4 meaningful player choices. Minimum 3 enforced by validation.
imagePromptstringYesDetailed image generation prompt (see §5)
reuseImagebooleanYesfalse by default. true only if visual setting is identical.
presentEntityIdsstring[]YesEntity IDs present/relevant in the scene
sceneSummarystringYes1-3 sentence summary for future context compression
entityUpdatesobjectNoKeyed by entity ID: { stateNote, profilePatch }. Must reflect plannedStateChanges.

4.3 Cross-Scene Analysis Propagation

The previous scene's flow analysis is passed forward into the next scene's context via the [PREVIOUS SCENE AI ANALYSIS] section in the user message. This creates a feedback loop:

  1. Scene N generates flow analysis fields alongside its content
  2. The raw AI response JSON is stored on the scene node as aiResponse
  3. When building context for Scene N+1, the system extracts the 5 analysis fields from Scene N's aiResponse
  4. These are included as [PREVIOUS SCENE AI ANALYSIS] in the user message, right before the player action
  5. This lets the model build on its prior reasoning — e.g. if satisfaction was low, it can course-correct
Storage: The complete raw JSON response (including flow analysis) is stored on each scene node's aiResponse field. This can be viewed and edited via the "View AI Response" button in the Scene Inspector (Content tab) or the brain icon debug button in play mode.
Schema enforcement: Gemini uses responseMimeType: "application/json" + responseSchema which guarantees valid JSON conforming to the schema — all 5 analysis fields + all scene content fields are enforced. OpenAI-compatible uses response_format: { type: "json_object" } which guarantees valid JSON but doesn't enforce the schema — field validation happens client-side.

5. Image Prompt Requirements

The imagePrompt field drives AI image generation. The system prompt instructs the model to:

  • Include the project's art style (from [ART STYLE] section)
  • Describe characters by physical appearance from their entity profiles (hair, clothing, build)
  • Include entity IDs in brackets, e.g. "a tall woman with silver hair [entity_abc123]"
  • Describe environment, lighting, mood, cinematic composition

5.1 Reference Image Collection

Before calling the image generation API, the system collects entity reference images for visual consistency:

  1. Gather entity IDs from the current scene's linked fields (linkedCharacters, linkedLocations, linkedObjects, linkedConcepts)
  2. Gather entity IDs from the new scene's presentEntityIds (returned by the model)
  3. For each entity ID, check if the entity has a referenceImage property
  4. Deduplicate (each entity's image is sent only once)
  5. Send all collected reference images to the image generation API
Assertion: If no reference images are found, a warning is logged listing each checked entity and whether it has a reference image. This helps diagnose missing reference images.

Additionally, a user-configured default image style (from AI Settings) is appended to every image prompt as a Style: suffix.

6. Validation & Assertions

6.1 Choice Validation

  • Minimum 3 choices enforced
  • If the model returns fewer than 3, generic fallback choices are appended: "Look around carefully", "Continue forward", "Consider your options"
  • Warning logged when padding occurs

6.2 Entity Tracking Validation

  • If presentEntityIds is empty or missing, a warning is logged
  • Present entity IDs are logged for debugging
  • Entities are categorized into linkedCharacters, linkedLocations, linkedObjects, linkedConcepts on the new scene node

6.3 JSON Parse Robustness

3-stage parsing pipeline:

  1. Direct parse: JSON.parse(response)
  2. Sanitized parse: Strip trailing commas and comments, then JSON.parse
  3. Regex fallback: Extract individual fields via regex patterns (last resort)

Additionally, legacy marker format ([SCENE_TEXT] + <<<OW_META>>>) is still supported for backward compatibility.

6.4 Image Generation

  • imagePrompt is required in the schema
  • If missing (parse failure), a fallback prompt is constructed from scene text + entity names
  • reuseImage must be explicitly true AND a current scene image must exist; otherwise, a new image is generated
  • Image generation has 3 retries with exponential backoff and prompt simplification

8. System Prompt & Instruction

8.1 System Prompt

Stored in WriterSettings.systemPrompt. Editable in AI Settings → Story Writer section. Defines:

  • Narrator role and personality
  • Writing style (screenplay-inspired)
  • Content guidelines
  • Output format (JSON fields and their requirements)
  • Rules for choices, entities, images, music

Click "Reset to Default" in AI Settings to restore the built-in system prompt.

8.2 Instruction

Stored in WriterSettings.instruction. Inserted at the END of the user message, right after the player's action. Tells the model:

Continue the story for another 100-300 words based on the player's action above. Consider all events, character states, variable conditions, and entity backgrounds from the scene timeline. Be emotionally intelligent, plausible, and interesting.

Also editable and resettable in AI Settings.
