Memory Optimization

Preventing OOM crashes in browser-based projects with many AI-generated images and audio.

The Problem

Each AI-generated scene image is 1-3 MB of base64 data stored as a JavaScript string on the V8 heap. A project with 20+ scenes creates hundreds of megabytes of heap pressure. With undo/redo history (up to 8 snapshots), this multiplies further. The browser tab eventually crashes with an OOM error.

Solution: Blob URL Cache + Asset Offloading

File: src/utils/blobCache.ts

Core Concept

Base64 data URL:  "data:image/png;base64,iVBOR..."  (~2 MB string on V8 heap)
                                    ↓
                            getBlobUrl(dataUrl)
                                    ↓
Blob URL:         "blob:http://localhost:5173/abc-123"  (~50 bytes on V8 heap)
                  + Blob object in native memory (outside V8, can page to disk)

~20,000x reduction per asset. The binary data lives in the browser's native blob storage, managed by the OS memory manager, not by V8's garbage collector.

Two Caches

CacheTypePurpose
cacheMap<fingerprint, blobUrl>Deduplicates data URLs by fingerprint (prefix + length + tail). Ensures same image always gets same blob URL.
blobStoreMap<blobUrl, Blob>Keeps Blob references alive. Without this, blob URLs become invalid after the Blob is garbage collected. Also used for rehydration back to base64.

Asset Offloading Flow

  1. New OW scene created with base64 images/audio on node data
  2. saveAndOffloadAssets() called after saving to IndexedDB
  3. collectAssetReplacements(project) scans all nodes + entities for base64 data URLs
  4. Each base64 string is converted to a Blob → blob URL via getBlobUrl()
  5. Zustand store is updated (via Immer) to replace base64 strings with blob URLs
  6. Result: ~2 MB string replaced with ~50 byte blob URL pointer. Binary data in native memory.

Rehydration for Save/Export

Before writing to IndexedDB or exporting to ZIP, blob URLs must be converted back to base64 data URLs:

rehydrateForSave(project)
  → structuredClone(project)    // Cheap clone (~50 bytes per blob URL)
  → For each blob URL field:
      blobUrlToBase64(blobUrl)  // FileReader reads from cached Blob → base64
  → Return clone with base64 data URLs restored

Lean History Snapshots

The undo/redo system (MAX_HISTORY = 8) uses lean snapshots that strip large binary data:

Stripped Fields

  • backgroundImage — scene images
  • backgroundMusic — scene audio
  • voiceoverAudio — scene voiceover
  • referenceImage — entity portraits
  • aiResponse — raw AI JSON response
  • constructedContext — full context string
  • constructedSystemPrompt — system prompt

Stripped values are replaced with '__asset_stripped__' placeholder.

Rehydration on Undo/Redo

When the user performs undo/redo, the restored snapshot has stripped assets. rehydrateAssets() copies live asset data from the current (pre-undo) project back into the restored snapshot. This ensures images and audio are not lost when navigating history.

Combined with asset stripping, the history system achieves ~99% memory reduction compared to storing full project snapshots with binary assets.

Lazy Node Thumbnails

File: src/components/nodes/SceneNode.tsx

Scene node image thumbnails use IntersectionObserver with a 200px root margin. When a scene node is outside the visible React Flow viewport:

  • Shows a lightweight placeholder icon (no <img> element)
  • No image decoding occurs
  • When node scrolls within 200px of viewport, sets isVisible=true and renders the actual image
  • Once visible, stays visible (avoids flicker during drag/pan)
  • Image source uses getBlobUrl() — displays the blob URL, not raw base64

Post-Load Offloading

When a project is loaded from IndexedDB, it contains base64 data URLs (since they were rehydrated during save). After loading:

  1. loadProject(id) loads from IndexedDB
  2. After a 100ms setTimeout (let React render first), collectAssetReplacements() runs
  3. All base64 data URLs are converted to blob URLs
  4. Store is updated with blob URLs

Reference Image Rehydration for Image Gen

Entity reference images used for Gemini image generation must be base64 data URLs (the API requires inline base64 data). After offloading, these are blob URLs. The fix:

openWorldService.ts:
  For each entity reference image:
    if (img.startsWith('data:'))  → use directly
    if (img.startsWith('blob:'))  → await blobUrlToBase64(img) → use result

Same fix applied in gameStateAPI.ts for the generate_scene_image command.

    ↓ setTimeout(100ms)

    ↓

    ↓

    ↓

    ↓

    ↓
