Open World Mode

AI-powered dynamic scene generation. Each player action generates a new scene with narrative text, choices, background image, and music.

Pipeline Overview

Player types action (or clicks generated choice)
    |
    v
1. Build Context (openWorldContext.ts)
   - Full story path from start node to current node
   - All entity profiles + state histories (untruncated)
   - All player actions / choices made
   - Per-scene entity state notes
   - Variables, art style, project notes
   - Previous AI analysis (chain-of-thought)
   - Writer instruction
    |
    v
2. Stream AI Response (vite.config.ts /api/open-world)
   - Gemini: structured JSON via responseSchema
   - OpenAI-compatible: json_object response_format
   - SSE streaming to browser
    |
    v
3. Parse Response (openWorldService.ts)
   - 3-stage JSON parse (direct → sanitized → regex fallback)
   - Extract: sceneText, choices, imagePrompt, entityUpdates, etc.
    |
    v
4. Display Pending Result
   - Scene text + choices shown immediately
   - Player sees "Continue to Next Scene" button
    |
    v
5. Background Tasks (non-blocking)
   ├── Image Generation (Gemini w/ entity reference images)
   └── Music Search (BM25 server, 3-8 keyword query)
    |
    v
6. Player clicks "Continue to Next Scene"
   → handleAdvanceToOwScene()
   → Creates new scene node + edge in graph
   → Applies entity updates (deferred, non-blocking):
     - stateNote on scene node
     - profilePatch on entity profile
     - stateChanges appended to entity.stateHistory
   → Applies variable changes
   → saveAndOffloadAssets() (base64 → blob URLs)

Context Construction

File: src/services/openWorldContext.ts

Philosophy

Include EVERYTHING in full detail by default. Only compress when the user-configurable token budget is exceeded. The AI sees all entity profiles, all state change histories, all player actions, all scene text — verbatim, never truncated — unless the context exceeds the budget.

Story Path Construction

The story path is built from the directed graph, not session history:

  1. Find project.settings.startNodeId (the designated start node)
  2. Run BFS shortest path (computeShortestScenePath()) from start to current node
  3. For each scene on the path, extract: storyText, speakerName, choiceMade (player action), entityStates, summary
  4. Player actions are resolved from: (a) playerAction field on next node, (b) edge choice label, (c) intermediate node tracing

Fallback: If no start node is set or current node is unreachable from start, falls back to session.history (ordered list of visited node IDs).

Each scene in the story timeline now includes entity state notes (short annotations stored on SceneNode.data.entityStates). In full mode:

  Forest of Shadows [entity_xyz789]: Corruption spreading from central clearing

In compressed mode (over budget):

  5. [The Dark Forest] Brief summary → Player action: "Shield Elena" | Entity states: Elena: charmed; Forest: corruption spreading

Token Budget & Compression

  • Default budget: 500,000 tokens (configurable in AI Settings)
  • Estimation: 1 token ≈ 4 characters
  • Hard character limit: 6,000,000 chars (~1.5M tokens) safety valve

Compression only triggers when Phase 1 (full fidelity) exceeds the budget:

  1. Step 1: Drop full profiles of entities NOT in current scene (keep summaries)
  2. Step 2: Compress oldest story path scenes to summaries (keep at least 5 most recent in full)
Never compressed: Entity state histories, player actions, entity profiles (current scene), current scene text, AI analysis, notes, variables, art style, writer instruction.

AI Response Schema

Gemini uses responseMimeType: "application/json" with responseSchema. OpenAI-compatible uses response_format: { type: "json_object" }.

Required Fields

FieldTypePurpose
relevantEntityTraitsSTRINGChain-of-thought: which entity traits/profiles/state changes are relevant to this scene
playerGoalHypothesisSTRINGWhat does the PLAYER want to experience (action, puzzles, drama, etc.)
sceneIntentHypothesisSTRINGWhat is the player trying to achieve with THIS specific action
lastSatisfactionEstimateSTRINGHow many scenes ago did the player last get what they wanted
engagementStrategySTRINGShould this scene SATISFY or CHALLENGE the player
plannedStateChangesSTRINGWhich entities will change state and how. Must have corresponding entityUpdates.
sceneTextSTRINGNarrative continuation (100-300 words). Screenplay-inspired.
speakerNameSTRINGNarrator or main speaking character name
choicesARRAY<STRING>Exactly 3 meaningful player choices
imagePromptSTRINGDetailed image gen prompt with entity IDs in brackets
sceneSummarySTRING1-3 sentence summary of key events
presentEntityIdsARRAY<STRING>Entity IDs present/relevant in this scene
reuseImageBOOLEANtrue only if visual setting is identical to previous scene

Optional Fields

FieldTypePurpose
entityUpdatesOBJECTKeys: entity IDs. Values: {stateNote, profilePatch, stateChanges}. Required when entities change state.
variableChangesOBJECTVariable name → new value pairs
newEntitiesARRAYNew entities to create: {category, name, description, summary?, profile?}
removeEntitiesARRAY<STRING>Entity IDs to permanently delete (death, destruction)
entityLinksARRAY<STRING>Entity IDs to link to scene (referenced but not physically present)
newVariablesARRAYNew tracking variables: {name, type, defaultValue?, description?}

generateVoiceoverBOOLEANAuto-generate TTS for this scene
musicQuerySTRING3-8 keywords for BM25 music search. Required for first scene and mood changes.

Analysis Fields (Chain-of-Thought)

The 6 analysis fields (relevantEntityTraits through plannedStateChanges) are placed BEFORE scene content in the schema. This forces the model to reason about entities, player goals, and engagement strategy before writing the scene text. The previous scene's analysis is propagated as [PREVIOUS SCENE AI ANALYSIS] in the next context.

Application Flow

  1. updateEntityState(newNodeId, entityId, stateNote) — Writes brief note to SceneNode.data.entityStates[entityId] (per-scene annotation)
  2. updateEntity(entityId, { profile }) — Merges profilePatch into entity's permanent profile
  3. updateEntity(entityId, { stateHistory }) — Appends new EntityStateChangeEvent with scene label, timestamp, player action, and detailed change descriptions
freshProject reference: The deferred entity update block captures useProjectStore.getState() once. Since each entity update modifies a different entity, there are no stale-read conflicts within the loop. The store uses Immer, so each updateEntity() call operates on the latest Zustand state.

Image Generation

Scene Image Generation

  1. Collect entity IDs from current scene's linked entities (all 4 categories)
  2. Collect entity IDs from AI's presentEntityIds (new scene)
  3. For each entity with a referenceImage, collect it
  4. Resolve blob URLs to base64: After asset offloading, referenceImage may be a blob URL. Use blobUrlToBase64() to convert back before sending to API.
  5. Send reference images to Gemini with prompt: "Using the reference images above for visual consistency, generate a new image: [prompt]"

Entity Portrait Generation (with Scene Reference)

When new entities are introduced or existing entities lack a reference image, a 512x512 portrait is generated automatically. The scene's background image is passed as a reference to the image generation API (Gemini only) so the portrait matches the art style and visual context of the introduction scene.

Sources

  1. AI-requested: The generateEntityImages field in the AI response explicitly requests images for specific entities with custom prompts.
  2. Auto-fallback: For any linked entity (character, location, object, concept) that lacks a referenceImage, a portrait is generated automatically using the entity's profile data (appearance, description).

Scene Image as Reference

  • The scene's background image is converted from blob URL to base64 via blobUrlToBase64().
  • Passed in the referenceImages array to /api/generate-image.
  • Gemini uses this for visual/style consistency — the entity portrait matches the scene's art style.
  • For non-Gemini providers, only the text prompt is used (no reference image support).

Category-Specific Prompts

CategoryPrompt Template

Concept"Visual representation of [name]. [description]. Abstract or symbolic illustration, 512x512 pixels."

Image Gen with Retry

  • Up to 3 attempts with progressive prompt simplification
  • Attempt 1: full prompt
  • Attempt 2: remove violence/weapons keywords
  • Attempt 3: keep only setting + mood

reuseImage Flag

AI sets reuseImage: true when the visual setting is identical to the previous scene (e.g., same room, no new characters). Skips image generation entirely — previous scene's image is reused.

Scene Transition Crossfade

When a scene's background image changes, a 2-second crossfade smoothly transitions from the old image to the new one.

  • Duration: 2 seconds (FADE_DURATION = 2000)
  • CSS: transition: opacity 2s ease-in-out on the current image layer
  • Layers: Two overlapping <img> elements — previous image at full opacity underneath, new image fading from 0 to 1 on top
  • First image: If no previous image exists, the new image fades in from 0 to 1
  • Image removed: Previous image stays visible (never goes to blank)
  • Cleanup: Previous image DOM element removed 2.2 seconds after transition starts

Music Integration

BM25 Music Server

  • FastAPI on port 7862 (rpg-music-server/bm25_server.py)
  • 2,580 tracks, 18 RPG genres
  • 3 search fields: situations, emotions, captions
  • Vite proxy: /api/music/* → localhost:7862/api/*

Search Flow

  1. AI provides musicQuery (3-8 keywords, e.g., "dark forest, mysterious, tense")
  2. POST /api/music/search with search_field: 'situations', top_k: 3, singing_filter: 'no_singing'
  3. Fetch top result metadata: GET /api/music/track/{rowId}
  4. Download audio, convert to base64 data URL
  5. Return to player for crossfade playback

TTS (Text-to-Speech)

File: src/services/ttsService.ts

Chunked Streaming

  1. Split scene text into growing chunks: ~100 chars (chunk 1), ~200 (chunk 2), ~400+ (subsequent)
  2. Split on sentence boundaries ([.!?]\s+ or \n\n)
  3. Send all chunks to POST /api/generate-tts in parallel
  4. Deliver results in order (not FIFO) using indexed array + delivery pointer
  5. Concatenate all chunks into single audio data URL for storage on scene node

API Endpoint

  • Model: gemini-2.5-flash-preview-tts (configurable)
  • Voice: configurable (e.g., "Zephyr")
  • Instruction: configurable (e.g., "Read aloud in a natural audiobook narrator style")
  • Returns: MP3 data URL per chunk

Context Viewer (Play Mode UI)

Bottom-left of play mode, next to the brain icon (AI Response viewer). Click the FileText icon to open a modal showing:

Auto-Resume from Save

When entering Open World mode without explicit startNode/mode URL params:

  1. Check all save slots for saves matching this project ID
  2. If saves exist: continue from the most recent save's node (preserving variables, inventory, flags, history)
  3. If the saved node no longer exists in the project: fall back to start node
  4. If no saves at all: start from the start node (fresh game)

Default Dream Room

Every new adventure is created with a default "Dream Room" start node — a white holodeck-like simulation room with introductory narration. This ensures players can immediately click the pink Open World button and start playing without any setup.

  • Background image: /dreamroom.jpg (served from public/)
  • Node label: "The Dream Room"
  • Story text: ~400 words introducing the player to the concept of an infinite simulation
  • Default choice: "Begin the adventure"

Start Node

  • Designated via "Set as Start Node" button in SceneInspector
  • Only one start node per project (project.settings.startNodeId)
  • Indicated by green ring + green dot on SceneNode canvas visual
  • Used as the root for BFS story path computation
  • Used as the starting scene for "Play from Start"
  • Used for auto-layout depth computation
  • Every new project automatically creates a start node ("The Dream Room") with a default background image and introductory text

Critical Implementation Details

Stale Project Bug (Fixed): generateOpenWorldScene() was previously called with the player store's project — a stale snapshot from game start that didn't include newly created OW nodes/edges. Fixed by using useProjectStore.getState().currentProject (fresh project store).
Blob URL Reference Images (Fixed): After asset offloading, entity.referenceImage holds blob URLs (~50 bytes) instead of base64 data URLs. The image gen API at /api/generate-image silently skipped blob URLs (checking startsWith('data:')). Fixed by resolving blob URLs back to base64 via blobUrlToBase64() before sending.
JSON not streamable: Since the AI outputs structured JSON (not free text), text streaming to the UI is not possible. The player sees a loading state during generation, then the full scene text appears at once with "Continue to Next Scene".
History suppression: All OW scene creation work (node creation, entity updates, variable changes) is wrapped in suppressHistoryRecording() to avoid recording intermediate states in undo history. Only the final state is captured.
