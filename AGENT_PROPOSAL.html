1. Executive Summary

Dream-E currently has two separate agent contexts that can manipulate game state:

Chat Agent CHAT

Uses claude -p --resume via Vite server middleware. Has access to all 56+ Game State API commands via <<<SW_CMD>>> blocks. Operates in an agentic loop (up to 10 iterations). Can chain multi-step workflows.

Open World Storyteller OW

Direct API calls to Gemini / OpenAI-compatible endpoint. Returns structured JSON with scene text + metadata. Currently has limited game state manipulation — only via entityUpdates, variableChanges, and presentEntityIds fields in the JSON response.

This proposal covers three initiatives:
A. Give the OW storyteller LLM full entity CRUD + profile patching + state management via its JSON output
B. Replace the Chat agent's Claude Code SDK dependency with direct LLM API calls (same model as OW writer)
C. Unify and expand the full set of agent operations across both contexts

2. Current Architecture

2.1 Chat Agent Pipeline

Browser Chat UI → POST /api/chat (SSE) → Vite Middleware → claude -p --resume <sid> → SSE stream back

Agent text + <<<SW_CMD:action>>> blocks → parsed in browser → executeCommand() → results sent back → next iteration.

2.2 Open World Pipeline

Player Action → buildOpenWorldContext() → POST /api/open-world (SSE) → Gemini/OpenAI API → JSON parse → Scene creation + image + music

OW model returns structured JSON with sceneText, choices, imagePrompt, variableChanges, presentEntityIds, entityUpdates, musicQuery.

3.3 Processing Pipeline (Post-Parse)

After the OW JSON response is parsed, the application processes state changes in this order:

  1. Create new entities — newEntities → call create_entity for each, collect generated IDs
  2. Apply entity updates — entityUpdates → call patch_entity_profile + update_entity_state
  3. Remove entities — removeEntities → call delete_entity for each
  4. Create new variables — newVariables → call create_variable
  5. Apply variable changes — variableChanges → update runtime variable values
  6. Create scene node — with text, choices, links to present entities
  7. Link entities to scene — entityLinks + newly created entity IDs
  8. Generate images — scene image (imagePrompt) + entity images (generateEntityImages) in parallel
  9. Search music — musicQuery → BM25 search + download
Context Update Requirement: The OW system prompt and instruction must be updated to document these new JSON fields and when the model should use them. The responseSchema (Gemini structured output) must also be expanded.

4. Initiative B — Replace Chat SDK with Direct LLM API

4.1 Current Problem

  • Chat agent uses claude -p --resume <session_id> — requires Claude Code CLI installed + authenticated (OAuth login)
  • Tight coupling to Anthropic-specific tooling; can't use the user's configured writer model (Gemini, etc.)
  • Requires the Vite dev server middleware to spawn child processes
  • Session persistence relies on Claude Code's internal session management

4.2 Proposed Architecture

Browser Chat UI → POST /api/chat (SSE) → Vite Middleware → Gemini / OpenAI API (direct) → SSE stream back

Reuse the same writer settings (provider, model, API key, endpoint) from useImageGenStore.writer. The middleware becomes a thin proxy — same pattern as /api/open-world.

4.4 Chat Session History Management

The server middleware maintains a message history array per session:

[
  { role: "system", content: "<auto-generated system prompt from registry>" },
  { role: "user", content: "[Game State]\n...\n\nUser message here" },
  { role: "assistant", content: "I'll create that scene...\n<<<SW_CMD:create_scene>>>..." },
  { role: "user", content: "[Command Results]\n...\n[Updated Game State]\n..." },
  ...
]

On POST /api/chat-reset, the history array is cleared. Optionally, history could be persisted into project.chatMessages for cross-session continuity.

Backward Compatibility: The browser-side agentic loop (aiChatService.ts) remains nearly identical. Only the request payload changes — adding provider, model, apiKey, endpoint fields. The <<<SW_CMD>>> command format and execution loop stay the same.

7. Chat Agent — Expanded Capabilities After Direct API Migration

After migrating from claude -p to direct API calls, the chat agent retains all 56+ existing commands and gains:

CapabilityDetailsStatus
Model choiceUses whatever model the user configured in AI Settings (Gemini, GPT-4o, custom endpoints)NEW
No CLI dependencyNo need for Claude Code CLI installed — just an API keyNEW
Proper message rolesSystem prompt as system role, not embedded in first user messageUPGRADE
Structured output (optional)For Gemini: could use responseSchema to enforce command format instead of regex parsingNEW
Shared model between OW + ChatSingle configuration point; both agents use the same modelUPGRADE
Server-managed historyFull conversation history in middleware; cleaner session resumeUPGRADE
Trade-off: The current Claude Code agent has access to Claude's agentic capabilities (tool use, file reading, etc.) which a raw API call won't have. However, for Dream-E's use case (game state manipulation via structured commands), the <<<SW_CMD>>> pattern provides equivalent functionality without needing native tool use.

9. Implementation Roadmap

Phase 1 — OW Entity Lifecycle

  1. Expand OWMetadata interface with newEntities, removeEntities, entityLinks, newVariables, generateEntityImages
  2. Update parseOpenWorldResponse() to extract new fields
  3. Add post-parse pipeline in generateOpenWorldScene() — entity creation → updates → deletion → variable creation → scene node creation → entity linking → image gen
  4. Update Gemini responseSchema in server middleware to include new fields
  5. Update DEFAULT_WRITER_INSTRUCTION and DEFAULT_WRITER_SYSTEM_PROMPT to document new fields
  6. Update OW_CONTEXT_MANUAL.html

Phase 2 — Chat Agent Direct API Migration

  1. Create new server middleware handler that calls Gemini / OpenAI API directly (reuse OW endpoint pattern)
  2. Add server-side message history management (array of messages per session)
  3. Update aiChatService.ts to send provider config with each request
  4. Update POST /api/chat handler to use direct API instead of claude -p
  5. Keep <<<SW_CMD>>> parsing + agentic loop unchanged in browser
  6. Deprecate claude -p middleware code

Phase 3 — Polish & Edge Cases

  1. Auto-TTS for OW scenes (generateVoiceover field)
  2. Entity image generation from OW (generateEntityImages)
  3. Structured output for chat commands (optional: use Gemini responseSchema to enforce command format)
  4. Save TTS audio to Asset Library in addition to scene node
  5. Cross-session chat history persistence
